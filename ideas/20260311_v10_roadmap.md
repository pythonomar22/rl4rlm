# V10 Roadmap: Final Training Run

Date: 2026-03-11

## Summary of All Learnings

After V1-V9, we know exactly what works and what doesn't.

## What Works

1. **Standard GRPO (V4-s5)**: Produces the best root code generation. 5 steps is the sweet spot — more steps lead to overfitting.
2. **Hybrid architecture**: Trained root + base sub-calls. Essential for multi-hop reasoning.
3. **SC-GRPO**: Eliminates mode collapse (0% vs 60%). BUT: V4-s5 (standard GRPO) + hybrid outperforms V7 (SC-GRPO) + hybrid. This suggests SC-GRPO may not be needed when hybrid prevents sub-call degradation.
4. **PPO loss**: Clipped IS ratio prevents gradient explosions.
5. **Raw advantages**: No std normalization, following cookbook and Dr. GRPO.
6. **Fixed sub-temperature (0.3)**: Prevents noisy sub-call extraction.
7. **Seed separation**: Training/eval seed offset of +100000.
8. **Anti-shortcut training**: Min 50K context forces genuine chunking.

## What Doesn't Work

1. **More than 5 steps**: All models show diminishing returns or regression after step 5
2. **SC-GRPO + hybrid**: Redundant — strategy diversity helps non-hybrid more than hybrid
3. **OOLONG-style tasks**: Model has never seen unstructured D&D transcripts; 0% on all configs
4. **Notebook QA in hybrid mode**: Base sub-calls can't parse Jupyter structure as well as trained sub-calls

## V10 Strategy

### Option A: More Training Data Diversity (Recommended)

Current training tasks: code_debug, cross_doc_compare, notebook_qa, dataframe_qa, hard_multi_hop, multi_hop_qa, event_counting, doc_classify, niah, multi_niah

**Missing**: OOLONG-style unstructured aggregation tasks.

Create synthetic OOLONG tasks:
- D&D-style narrative text with embedded counts
- News articles with embedded statistics
- Meeting transcripts with action items to count
- Any text where counting requires understanding natural language, not just pattern matching

### Option B: Curriculum Learning

Step 1-2: Easy tasks (NIAH, Multi-NIAH, Doc-Classify) — learn the API
Step 3-5: Medium tasks (Multi-Hop, Event Counting, Code Debug) — learn decomposition
Step 6+: Hard tasks (Hard Multi-Hop, Cross-Doc, OOLONG) — learn chaining

### Option C: Self-Play Improvement

1. Run V4-s5-hybrid on all training tasks
2. Collect successful trajectories
3. SFT on successful trajectories to distill hybrid behavior into a single model
4. RL from the SFT checkpoint

This addresses the root problem: we want a single model that generates both good root code AND good sub-call responses.

### Option D: Adapter Gating (Most Novel)

Train two LoRA adapters:
- Adapter A: Root code generation (activated for assistant turns)
- Adapter B: Sub-call response (activated for llm_query() turns)

This architecturally separates planning from perception, achieving hybrid behavior without needing two model instances at inference.

## Recommended V10 Config

If V9 at step 5 shows PPO+raw advantages produce stable training:
- Start from V9-s5 (or V4-s5 if V9 isn't better)
- Continue with hybrid architecture for inference
- Add OOLONG-style training tasks
- Use curriculum (easy→hard over steps)
- Evaluate at step 5, 10, 15
- Best checkpoint becomes final model

## Final Evaluation Plan

For the paper, run ALL configurations on ALL benchmarks:
1. Base model
2. V4-s5 (standard GRPO)
3. V4-s5-hybrid
4. V9-s5
5. V9-s5-hybrid
6. V10-s5-hybrid (final)

Each with identical seeds, 20 tasks per benchmark, 11+ benchmarks.
